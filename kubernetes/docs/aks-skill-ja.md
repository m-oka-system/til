# Azure Kubernetes Service

> **権威あるガイダンス — 必須準拠**
>
> このスキルは、ユーザー要件に基づいた **推奨 AKS クラスター構成** を生成します。**Day-0 決定事項**（ネットワーク、API サーバー — 後から変更が困難）と **Day-1 機能**（作成後に有効化可能）を区別します。CLI コマンドについては [CLI リファレンス](./references/cli-reference.md) を参照してください。

## クイックリファレンス

| プロパティ | 値                                                                                  |
| ---------- | ----------------------------------------------------------------------------------- |
| 最適な用途 | AKS クラスターの計画と Day-0 決定事項                                               |
| MCP ツール | `mcp_azure_mcp_aks`                                                                 |
| CLI        | `az aks create`、`az aks show`、`kubectl get`、`kubectl describe`                   |
| 関連スキル | azure-diagnostics（AKS トラブルシューティング）、azure-validate（準備状況チェック） |

## このスキルを使用するタイミング

以下のことをユーザーが望む場合にこのスキルを有効化します:

- 新しい AKS クラスターの作成
- 本番ワークロード向けの AKS クラスター構成の計画
- AKS ネットワーキングの設計（API サーバーアクセス、Pod IP モデル、エグレス）
- AKS の ID とシークレット管理のセットアップ
- AKS ガバナンスの構成（Azure Policy、Deployment Safeguards）
- AKS オブザーバビリティの有効化（Container Insights、Managed Prometheus、Grafana）
- AKS のアップグレードとパッチ戦略の定義
- AKS のコスト可視性と分析の有効化
- AKS Automatic と Standard SKU の違いの理解
- AKS クラスターのセットアップと構成の Day-0 チェックリストの取得

## ルール

1. コンピュート、ネットワーキング、セキュリティ、その他の設定のプロビジョニングに関するユーザー要件から開始する。
2. `azure` MCP サーバーを使用し、まず `mcp_azure_mcp_aks` を選択して、クライアントが公開する正確な AKS 固有の MCP ツールを検出する。タスクに適合する最小の検出済み AKS ツールを選択し、必要な機能が AKS MCP サーフェスを通じて公開されていない場合のみ Azure CLI（`az aks`）にフォールバックする。
3. ユーザーの制御とコンビニエンスの要件に基づいて、AKS Automatic と Standard SKU のどちらが適切かを判断する。特定のカスタマイズが必要でない限り、デフォルトは AKS Automatic とする。
4. クラスター構成の選択に関する決定と根拠を文書化する。特に後から変更が困難な Day-0 決定事項（ネットワーキング、API サーバーアクセス）について。

## 必須入力（必要なもののみ確認）

ユーザーが不明な場合は、安全なデフォルト値を使用する。

- AKS 環境タイプ: dev/test または production
- リージョン、アベイラビリティゾーン、希望するノード VM サイズ
- 想定スケール（ノード/クラスター数、ワークロードサイズ）
- ネットワーキング要件（API サーバーアクセス、Pod IP モデル、イングレス/エグレス制御）
- セキュリティと ID 要件（イメージレジストリを含む）
- アップグレードとオブザーバビリティの設定
- コスト制約

## ワークフロー

### 1. クラスタータイプ

- **AKS Automatic**（デフォルト）: ほとんどの本番ワークロードに最適。セキュリティ、信頼性、パフォーマンスのベストプラクティスが事前構成されたキュレーテッドエクスペリエンスを提供する。Node Auto-Provisioning（NAP）でサポートされないネットワーキング、オートスケーリング、ノードプール構成に特定のカスタム要件がない限り使用する。
- **AKS Standard**: 環境構成を完全に制御する必要がある場合に使用する。セットアップと管理に追加のオーバーヘッドが必要。

### 2. ネットワーキング（Pod IP、エグレス、イングレス、データプレーン）

**Pod IP モデル**（重要な Day-0 決定事項）:

- **Azure CNI Overlay**（推奨）: プライベートオーバーレイ範囲からの Pod IP。VNet ルーティング不可。大規模環境にスケール可能で、ほとんどのワークロードに適合。
- **Azure CNI（VNet ルーティング可能）**: VNet から直接 Pod IP を取得（Pod サブネットまたはノードサブネット）。Pod が VNet またはオンプレミスから直接アドレス可能である必要がある場合に使用。
  - ドキュメント: https://learn.microsoft.com/azure/aks/azure-cni-overlay

**データプレーンとネットワークポリシー**:

- **Azure CNI powered by Cilium**（推奨）: 高性能パケット処理、ネットワークポリシー、オブザーバビリティのための eBPF ベース

**エグレス**:

- 安定した予測可能なアウトバウンド IP のための **Static Egress Gateway**
- エグレス制限: UDR + Azure Firewall または NVA

**イングレス**:

- **App Routing アドオン + Gateway API** — HTTP/HTTPS ワークロードの推奨デフォルト
- **Istio サービスメッシュ + Gateway API** — 高度なトラフィック管理、mTLS、カナリアリリース向け
- **Application Gateway for Containers** — WAF 統合による L7 ロードバランシング向け

**DNS**:

- 信頼性が高くパフォーマンスの良い DNS 解決のために、すべてのノードプールで **LocalDNS** を有効化

### 3. セキュリティ

- **Microsoft Entra ID** をすべてで使用（コントロールプレーン、Pod の Workload Identity、ノードアクセス）。静的な資格情報は避ける。
- **Secrets Store CSI Driver** 経由の Azure Key Vault でシークレット管理
- **Azure Policy** + **Deployment Safeguards** を有効化
- etcd/API サーバーの**保存時暗号化**、ノード間の**転送中暗号化**を有効化
- 署名済みでポリシー承認済みのイメージのみを許可（Azure Policy + Ratify）。**Azure Container Registry** を推奨
- **分離**: 名前空間、ネットワークポリシー、スコープ付きログを使用

### 4. オブザーバビリティ

- AKS オブザーバビリティ（ログ + メトリクス）には Managed Prometheus と Container Insights + Grafana を使用する。
- セキュリティ監視とトラブルシューティングのために、Log Analytics ワークスペースでコントロールプレーンログと監査ログを収集する Diagnostic Settings を有効化する。
- その他の監視およびトラブルシューティングツールには、AKS 向け Agentic CLI、Application Insights、Resource Health Center、AppLens デテクター、Azure Advisors などの機能を使用する。

### 5. アップグレードとパッチ

- 制御されたアップグレードタイミングのために **Maintenance Windows** を構成
- セキュリティパッチと Kubernetes バージョンを最新に保つため、コントロールプレーンとノード OS の**自動アップグレード**を有効化
- エンタープライズ安定性のために **LTS バージョン**を検討（2 年間サポート）。AKS 環境を Premium ティアにアップグレード
- **フリートアップグレード**: テストから本番環境への段階的ロールアウトには **AKS Fleet Manager** を使用

### 6. パフォーマンス

- より高速なノード起動のために **エフェメラル OS ディスク**（`--node-osdisk-type Ephemeral`）を使用
- ノード OS として **Azure Linux** を選択（フットプリントが小さく、起動が高速）
- HPA を超えるイベントドリブンオートスケーリングのために **KEDA** を有効化

### 7. ノードプールとコンピュート

- **専用システムノードプール**: 最低 2 ノード。システムワークロード専用にテイント設定（`CriticalAddonsOnly`）
- コスト削減と応答性の高いスケーリングのために、すべてのプールで **Node Auto Provisioning（NAP）** を有効化
- ホストレベルの最適化のために**最新世代 SKU（v5/v6）**を使用
- **B シリーズ VM は避ける** — バースタブル SKU はパフォーマンス/信頼性の問題を引き起こす
- 本番ワークロードには**最低 4 vCPU** の SKU を使用
- SLO に基づいてホスト/ゾーン間で Pod を分散するために**トポロジー分散制約**を設定

### 8. 信頼性

- **3 つのアベイラビリティゾーン**にデプロイ（`--zones 1 2 3`）
- ゾーン冗長コントロールプレーン + API サーバー可用性 99.95% SLA のために **Standard ティア**を使用
- ランタイム保護のために **Microsoft Defender for Containers** を有効化
- すべての本番ワークロードに **PodDisruptionBudgets** を構成
- 障害ドメイン間の Pod 分散を確保するために**トポロジー分散制約**を使用

### 9. コスト管理

- バッチ/中断可能なワークロードには **Spot ノードプール**を使用（最大 90% 節約）
- dev/test クラスターの**停止/起動**: `az aks stop/start`
- 定常状態のワークロードには **Reserved Instances** または **Savings Plans** を検討

## ガードレール / 安全性

- シークレット（トークン、キー）を要求または出力しない。
- Day-0 の重要な決定事項で要件が曖昧な場合は、ユーザーに明確化のための質問をする。Day-1 有効化機能については、トレードオフを含む 2〜3 の安全なオプションを提案し、保守的なデフォルトを選択する。
- ゼロダウンタイムを約束しない。ワークロードのセーフガード（PDB、プローブ、レプリカ）と段階的アップグレード、および信頼性とパフォーマンスのベストプラクティスを推奨する。

## MCP ツール

| ツール              | 目的                                                                                   | 主なパラメータ                                                          |
| ------------------- | -------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| `mcp_azure_mcp_aks` | クライアントが公開する正確な AKS 固有ツールを検出するための AKS MCP エントリーポイント | まず呼び出し可能な AKS ツールを検出し、そのツールのパラメータを使用する |

## エラーハンドリング

| エラー / 症状                              | 原因の可能性                                                | 対処法                                                                                                                                                |
| ------------------------------------------ | ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| MCP ツール呼び出しの失敗またはタイムアウト | 無効な資格情報、サブスクリプション、または AKS コンテキスト | `az login` を確認し、`az account show` でアクティブなサブスクリプションコンテキストを確認。サブスクリプション識別子をユーザーにエコーバックしないこと |
| クォータ超過                               | リージョンの vCPU またはリソース制限                        | クォータの引き上げを要求するか、別のリージョン/VM SKU を選択                                                                                          |
| ネットワーキング競合（IP 枯渇）            | Overlay/CNI に対して Pod サブネットが小さすぎる             | IP 範囲を再計画。クラスターの再作成が必要な場合あり（Day-0）                                                                                          |
| Workload Identity が動作しない             | OIDC イシュアーまたはフェデレーテッド資格情報の欠落         | `--enable-oidc-issuer --enable-workload-identity` を有効化し、フェデレーテッド ID を構成                                                              |
